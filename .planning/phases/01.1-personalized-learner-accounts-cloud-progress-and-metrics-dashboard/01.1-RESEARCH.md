# Phase 01.1 Research: Personalized Learner Accounts, Cloud Progress, and Metrics Dashboard

**Researched:** July 19, 2026  
**Phase:** 01.1  
**Requirements:** PERS-01, PERS-02, AUTH-01, AUTH-02, CLOUD-01, CLOUD-02, PRIV-01, SEC-03, QA-03, DEP-05

## Executive conclusion

The requested expansion is feasible before the submission deadline only as an optional, configuration-gated layer over the existing local experience. The lowest-risk architecture is:

1. calculate personal metrics locally from strict learning evidence first;
2. keep every challenge route public and guest-capable;
3. use the Firebase browser SDK only for optional Google Authentication;
4. send the Firebase ID token to same-origin Next.js Route Handlers;
5. verify identity and perform Firestore access inside a `server-only` data-access layer;
6. write progress from the authoritative assessment lifecycle, not from a browser-declared completion;
7. test Firebase Auth and Firestore behavior against a demo-project Emulator Suite before any real project is configured;
8. enable real cloud sync and Netlify preview only after explicit user credential/deployment approval;
9. retain a single configuration switch that returns the application to local-only behavior.

This structure adds visible longitudinal value while preserving the already-green fixture fallback and judging access. Direct browser writes to Firestore are not recommended because Security Rules can isolate UIDs but cannot independently prove FaultSmith's deterministic assessment semantics.

## Current implementation seams

### Browser state

`src/components/faultsmith-app.tsx` currently owns most product state. It restores and writes:

- `faultsmith:attempt:v2` for public challenge, learner source/prose, and report restoration;
- `faultsmith:events:v1` for a capped 100-event anonymous log without learner prose;
- `faultsmith:learning-progress:v1` for at most nine strict completion records.

The learning record already contains `stepId`, `completedAt`, `overallScore`, `hintsUsed`, and `testRuns`. This is enough for a useful first dashboard. It is not enough for dimension trends or recent attempts, so a separate bounded attempt-summary schema is required. The source/prose-rich active attempt must never be reused as a cloud DTO.

### Learning policy

`src/lib/learning-paths.ts` already provides strict progress parsing, duplicate replacement, ordered unlocks, and deterministic recommendations. It records completion only after the assessment response says `verified`. Phase 01.1 should extend this domain through new metric/merge helpers rather than embedding calculations in React.

### Assessment authority

`src/app/api/challenges/assess/route.ts` validates a strict request, calls `assessChallengeWorkflow`, and returns a no-store result. The client currently records guided completion afterward. For server-verified cloud provenance, this route is the correct integration point: derive the approved lesson from the server-owned fixture/challenge mapping, verify an optional Firebase token, run the existing assessment, and persist only the bounded summary after deterministic policy has decided status.

Cloud failure must not change the assessment response to failure. Persistence should return a bounded sync status or be attempted through a service that can safely report `local_only`/`cloud_degraded` while the learning report remains authoritative.

### Request protection

`src/server/request-guard.ts` already enforces JSON content type, streamed 80 kB caps, sanitized errors, per-scope and per-IP in-memory buckets, and bounded bucket growth. Phase 01.1 needs a separate bounded Authorization parser and Firebase-error mapping. The raw bearer token must never reach logs or errors. In-memory limits remain defense in depth; Netlify path-level rules are still needed for public multi-instance control.

### Client request helper

The current `postJson` sends only JSON and has a 35-second abort. It needs an optional token-provider seam rather than global token state copied into localStorage. Firebase manages its own browser session; FaultSmith should request a fresh ID token for authenticated progress/assessment calls and keep token values out of application persistence.

### CSP and health

`next.config.ts` currently limits `connect-src` to same-origin (and development WebSockets), with no Firebase/Google auth origins. Google sign-in will fail until the exact observed Firebase origins are allowlisted. The health route reports only safe OpenAI/fallback booleans. It may add a coarse cloud mode but must not return project IDs, domains, UIDs, credential state detail, or errors.

## Platform findings

### Firebase Authentication

Firebase officially supports Google Authentication for web applications and optional anonymous-to-permanent account linking. FaultSmith should not auto-create anonymous Firebase accounts for every judge: guest localStorage already supplies anonymous persistence, avoids account quota/noise, and requires no external request. Start Firebase authentication only when the learner selects synchronization.

Use Google as the one persistent provider. Capture cancellation and popup-block errors as non-fatal UI states. Redirect fallback should be enabled only if tested because redirect state and authorized-domain behavior differ across local and preview origins.

Official guidance: [Google sign-in for web](https://firebase.google.com/docs/auth/web/google-signin) and [Firebase Authentication](https://firebase.google.com/docs/auth).

### Server token verification

Firebase's supported custom-backend flow is: retrieve an ID token from the signed-in client, send it to the backend over HTTPS, verify it with the Admin SDK, and use the verified UID. The server must not accept UID, document path, email, or provider claims from the request body.

The token verifier must distinguish only stable public outcomes (`AUTH_REQUIRED`, `AUTH_INVALID`, `AUTH_UNAVAILABLE`) and collapse provider-specific errors. Verification needs the correct Firebase project ID and rejects malformed, expired, wrong-audience/project, and invalid-signature tokens.

Official guidance: [Verify ID tokens](https://firebase.google.com/docs/auth/admin/verify-id-tokens).

### Firebase Admin runtime

Firebase's current Admin SDK setup documentation requires Node.js 22 or newer. FaultSmith CI already uses Node 24 and the developer workstation uses Node 24.9, but `package.json` currently permits Node 20.9. Phase 01.1 must align the declared engine and Netlify build/function runtime with a reviewed Node 22-or-newer choice before adding the current Admin SDK.

On a non-Google host such as Netlify, Admin access requires privileged service-account material. Do not generate or commit a JSON file in the repository. The implementation should parse one server-only host secret through a `server-only` initialization module, normalize multiline/private-key representation in memory, validate only required fields, and never serialize it. The user creates/configures this secret privately.

Official guidance: [Firebase Admin setup](https://firebase.google.com/docs/admin/setup).

### Cloud Firestore and free tier

Current Spark documentation supplies one free database per project, 1 GiB stored data, 50,000 reads/day, 20,000 writes/day, 20,000 deletes/day, and 10 GiB monthly outbound data. Quotas reset daily where documented; exceeding Spark limits can make the service unavailable rather than seamlessly billing. This makes local fallback mandatory.

Avoid paid/non-free features such as TTL deletion, PITR, backup/restore, clone, and Cloud Functions. Enforce retention in application code. A profile document plus capped attempt-summary subcollection is small enough for submission use if reads happen on sign-in/dashboard entry rather than every render.

Official guidance: [Firestore usage and limits](https://firebase.google.com/docs/firestore/quotas) and [Firebase pricing plans](https://firebase.google.com/docs/projects/billing/firebase-pricing-plans).

### Firestore authorization model

Firebase Security Rules can restrict client access by `request.auth.uid`, but server Admin libraries bypass rules. Since FaultSmith will use Admin for authoritative writes, the server-side repository and DTO contracts are the primary authorization boundary. Deploy default-deny Firestore Rules anyway so an accidentally imported browser Firestore client cannot read or write learning data.

Recommended hierarchy:

```text
learningProfiles/{verifiedUid}
learningProfiles/{verifiedUid}/attempts/{serverAttemptId}
```

Never accept collection, UID, or document names from the browser. Use a deterministic idempotency digest for the exact assessed snapshot/relevant bounded metadata, or a server-owned attempt ID with a uniqueness record. Do not include source/prose in stored idempotency material.

### Firebase Emulator Suite

Firebase recommends the Local Emulator Suite for Authentication/Firestore development and Security Rules testing. Use a `demo-` project ID so accidental non-emulated calls fail rather than touch live resources. Admin SDKs accept Auth emulator tokens only when the documented emulator environment variables are set; those variables must never be set in production.

Add a non-watch script such as:

```text
firebase emulators:exec --project demo-faultsmith --only auth,firestore "vitest run <emulator tests>"
```

The Firestore emulator needs Java. CI should set up a reviewed Java LTS runtime explicitly rather than depending on runner ambient state. Normal unit tests should still mock the repository and remain fast; emulator integration becomes a separately named gate or an explicit part of Unit and integration with adequate timeout.

Official guidance: [Local Emulator Suite](https://firebase.google.com/docs/emulator-suite), [Auth emulator](https://firebase.google.com/docs/emulator-suite/connect_auth), and [Firestore Rules tests](https://firebase.google.com/docs/firestore/security/test-rules-emulator).

### App Check

Firebase App Check can supplement abuse defense for web clients. Current documentation recommends reCAPTCHA Enterprise for new web integrations and describes a no-cost assessment allowance, but enforcement can block legitimate local, preview, or privacy-restricted browsers. Treat App Check as preview observation only in this phase. Do not represent it as a replacement for Auth, authorization, Security Rules, server validation, or rate limiting.

Official guidance: [App Check for web](https://firebase.google.com/docs/app-check/web/recaptcha-provider).

### Netlify and Next.js

Netlify currently supports major Next.js App Router and Route Handler behavior through OpenNext. The application remains a Node-backed Next.js deployment, not a static export. Public `NEXT_PUBLIC_` Firebase values are frozen into the client bundle at `next build`, so preview and production contexts need the correct public config at build time. Server Firebase/OpenAI credentials stay in the host environment and must never be in `netlify.toml`.

Netlify offers path/function rate-limiting capabilities on all plans, but adapter-generated Next Route Handlers may not map cleanly to hand-authored Netlify function config. Treat host UI/path rules plus deploy-log verification as the release path unless a reviewed explicit edge wrapper is added. Preserve application limits as defense in depth.

Official guidance: [Next.js on Netlify](https://docs.netlify.com/build/frameworks/framework-setup-guides/nextjs/overview/), [Netlify environment variables](https://docs.netlify.com/build/environment-variables/overview/), and [Netlify rate limiting](https://docs.netlify.com/manage/security/secure-access-to-sites/rate-limiting/).

### Next.js 16 constraints

Bundled Next.js 16 documentation confirms:

- Route Handlers are public endpoints and must perform their own authentication and authorization.
- A server-only Data Access Layer should centralize credential verification, database access, and minimal DTO construction.
- `server-only` imports should guard privileged modules.
- `NEXT_PUBLIC_` variables are inlined at build time and intentionally visible to the browser.
- Lambda-style deployments cannot share in-memory state across requests and may terminate long requests.
- GET Route Handlers are uncached by default but progress routes should still explicitly return `Cache-Control: no-store`.
- Nonce CSP requires dynamic rendering and has performance/hosting implications; Phase 01.1 should first minimize exact Firebase origin additions to the existing policy and separately re-evaluate nonce/hash migration rather than coupling it to auth delivery.

## Recommended code organization

Keep client-safe contracts/selectors separate from privileged modules:

```text
src/lib/progress-contracts.ts          strict cloud/local DTO schemas
src/lib/progress-metrics.ts            deterministic metrics and recommendations
src/lib/progress-merge.ts              provenance-aware monotonic merge
src/client/firebase-auth.ts            public config + optional Google token provider
src/server/firebase-admin.ts           server-only lazy initialization
src/server/firebase-identity.ts        bounded bearer parsing + verified UID
src/server/progress-repository.ts       server-only Firestore DAL
src/server/progress-service.ts          authorization, DTO, idempotency, retention
src/components/progress-dashboard.tsx   local/cloud personal evidence UI
src/components/progress-sync.tsx        sign-in/out/status/delete controls
src/app/api/progress/route.ts           GET/import/delete or bounded subroutes
firebase.json
firestore.rules                         deny direct browser access
```

Do not import Firebase Admin, service config, fixture hidden fields, or repository objects through `src/lib` modules consumed by the current client component.

## Data and metric recommendations

### Profile

Keep a single bounded profile with schema version, updated time, and at most nine completion entries. Each completion includes only approved lesson ID, time, overall score, hints, test runs, and `server_verified | local_import` provenance.

### Attempt history

Store at most 50 summaries. Useful fields already exist in the assessment response: four score dimensions, completion status, hints, test runs, changed lines, elapsed time, project, skill, difficulty, and source label. Bucket duration before persistence and omit exact learner content.

### Metric semantics

- Use verified attempts for score/skill averages.
- Show unverified attempts only as recent practice outcomes and reinforcement signals.
- Independent-solve rate means verified with zero hints.
- Do not turn test-run count or duration into a negative grade.
- Continue the existing `<80 or >1 hint` reinforcement rule initially.
- Prefer evidence cards/bars and plain text over a new chart dependency.

## Synchronization strategy

1. Restore strict local state immediately.
2. If cloud is disabled, render dashboard from local selectors only.
3. If learner signs in, obtain a fresh ID token and GET cloud DTO.
4. Merge completions by approved step and newest evidence, preserving provenance.
5. Optionally import bounded pre-existing local progress once as `local_import` with an idempotency marker.
6. On assessment, update local report/progress first; send the authenticated assessment request; server persists bounded result only after deterministic policy.
7. On transient cloud failure, keep local state, display degraded status, and allow a manual bounded retry.
8. Never background-loop indefinitely or sync on each render.

## Security/adversarial cases required in plans

- missing/duplicate/mixed-case/oversized Authorization headers;
- bearer token accidentally included in error/log/evidence;
- wrong Firebase project/audience, expired, revoked-shaped, malformed, unsigned emulator token in production;
- client-supplied UID, document path, unexpected DTO field, prototype key, unknown lesson, over-range scores/counts/times;
- cross-user read/import/delete and document-existence inference;
- replayed assessment/import, concurrent duplicate, out-of-order local/cloud writes;
- Firestore partial failure after assessment success;
- direct client Firestore access despite valid Firebase auth;
- service-account/private-key exposure through public env, source, Git history, error, server log, React serialization, or `.next/static`;
- public Firebase API key intentionally present but mistaken for server authorization;
- CSP overexpansion or auth flow breakage at localhost, preview, final domain, popup blocked, third-party cookies restricted;
- Firebase quota/unavailable timeout blocking the learning loop;
- cloud deletion removes another UID or silently claims success;
- disabling config causes import-time or build-time crashes;
- Netlify multi-instance bypass of in-memory rate buckets.

## Primary implementation risks and mitigations

| Risk | Mitigation |
|---|---|
| Scope destabilizes green candidate | Independent Phase 01.1 commits, feature configuration default-off until green, exact rollback switch |
| Client writes fake mastery | Persist future authoritative progress from assessment server path; label local imports |
| Cross-user data access | Verified UID-only paths, DAL, DTOs, emulator isolation tests, direct-client default deny |
| Service credential leaks | One server-only secret, `server-only` module, source/history/bundle tests, sanitized errors/logs |
| Firebase SDK breaks build when unset | Lazy complete-config guard; no module initialization at import when disabled |
| Auth CSP breaks judges | Exact origins only, localhost/preview/production browser proof, guest route always available |
| Free quota exhausted | Read on sign-in/dashboard, write on assessment/import only, 50-attempt cap, bounded retries, local fallback |
| Admin SDK runtime mismatch | Declare and deploy Node 22+; CI already Node 24 |
| Emulator test becomes slow/flaky | Fast mocked unit tier plus separate deterministic emulator command with Java setup and demo project |
| Demo exceeds three minutes | Dashboard gets a brief proof moment; full sign-in sync can be secondary evidence rather than primary lab path |

## Validation Architecture

### Test layers

1. **Fast domain/unit layer (every task):** strict schemas, metrics, merge, config detection, bearer parsing, repository/service behavior with fakes, error sanitation, idempotency, retention.
2. **Route/workflow integration (every implementation plan):** progress read/import/delete and assessment persistence with mocked identity/repository; exact response keys and no-store behavior; existing challenge routes remain public.
3. **Firebase emulator integration (after persistence boundary):** Auth token verification, Firestore same-user access, cross-user denial, direct-client rules denial, deletion, replay/concurrency, persistence across process calls.
4. **Browser/accessibility (after UI integration):** guest dashboard, signed-in mocked/emulated sync, cancellation/failure/sign-out/degraded state, refresh, keyboard, axe, 1440×900, 390×844.
5. **Security/release (each wave and final):** source/history and bundle leakage, dependency audit, production build, fallback/live smoke tooling, primary demo, independent QA/security review.
6. **Credentialed external proof (manual checkpoint):** real Firebase project locally, then explicit Netlify preview approval; guest and cross-session Google sync, CSP, logs, quotas, rate controls, config-off rollback. This test never runs in normal CI.

### Suggested commands

- Focused: `npx vitest run <changed test files>`
- Fast project: `npm run lint && npm run typecheck && npm test`
- Firebase emulator: a new non-watch `npm run test:firebase` using `firebase emulators:exec --project demo-faultsmith`
- Full: `npm run security:source && npm run quality && npm audit --audit-level=moderate`
- Release: existing fallback and production smoke plus new sanitized progress smoke.

### Feedback target

- Domain/route tests: under 30 seconds.
- Full non-emulator suite: preserve current under-two-minute target.
- Emulator suite: target under 90 seconds and run at least once per persistence wave and in CI.
- Credentialed Firebase/Netlify proof: manual, sanitized, SHA-bound, and separate from deterministic CI.

### Wave 0 needs

- Add Firebase SDK dependencies and a pinned/reviewed CLI/testing setup.
- Add `firebase.json`, default-deny `firestore.rules`, emulator ports, and demo project configuration.
- Add a Java LTS setup to the Firebase CI job.
- Add focused test fixtures/fakes that contain no real Firebase configuration.
- Extend security scanners before any credentialed proof.

### Nyquist mapping

Every task must have a focused automated check. No three consecutive implementation tasks may rely only on manual verification. Real Firebase and Netlify checks are explicit human checkpoints after the complete offline/emulator candidate passes; their absence cannot be substituted with mocks, and their failure must disable cloud mode rather than weaken a gate.

## Planning recommendation

Plan six bounded deliverables:

1. progress contracts, migration, metrics, and local dashboard;
2. Firebase dependencies/config/client auth plus emulator/CI foundation;
3. server identity DAL, Firestore repository, progress APIs, and assessment persistence;
4. sync UI, merge, deletion, degraded states, CSP, and browser coverage;
5. independent integration/security/QA hardening, documentation, and exact-SHA offline freeze;
6. human checkpoint for real Firebase configuration and separately approved Netlify preview proof, with configuration-off rollback.

Plans 1 and 2 can run in parallel only if they own disjoint files. Plan 3 depends on both. Plan 4 depends on domain and server contracts. Plan 5 integrates and self-heals. Plan 6 is non-autonomous and must not create accounts, credentials, or deployments on the user's behalf.
