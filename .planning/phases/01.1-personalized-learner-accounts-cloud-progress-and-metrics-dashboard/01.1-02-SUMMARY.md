# Phase 01.1 Plan 02: Firebase Foundation and Lazy Auth Client Summary

**Credential-free Firebase foundation: Node 22+ runtime with audit-clean firebase/firebase-admin/firebase-tools, a lazy browser-only email/password + Google auth adapter that stays local_only when unconfigured, default-deny Firestore rules proven in a demo-project emulator, and source/history/bundle scanners extended to fail closed on service-account, JWT, password-boundary, and Admin-SDK leakage.**

## Accomplishments

- Firebase web SDK 12, Admin SDK 14, pinned CLI 15.24.0, and rules-unit-testing installed with `npm audit --audit-level=moderate` reporting zero vulnerabilities; `engines.node >=22.0.0` and `.nvmrc 24` align with current Firebase Admin support while CI stays Node 24.
- `src/client/firebase-auth.ts` implements the full optional auth surface (subscription, create/login, `validatePassword`, verification/resend and reset with 60s local cooldowns, Google popup, explicit UID-preserving link with safe sign-out fallback, sign-out, fresh ID tokens) behind a lazily loaded gateway; importing the module never touches Firebase, and missing/partial/off config returns `local_only` without throwing at import, build, or runtime.
- Error collapsing is enumeration-resistant: unknown-user, wrong-password, and address-already-registered all share `credentials_rejected`; password reset reports generic success for unknown accounts; popup cancel/block and provider collisions are bounded non-fatal states; no error carries password, token, email, or provider payloads (asserted by tests).
- `firestore.rules` denies every direct browser read/write/list/delete; 11 emulator tests under the `demo-faultsmith` project prove denial for guest, unverified email, verified email, Google, linked-UID, collision-fallback, cross-user, and post-sign-out states against seeded documents.
- Source/history scanner now catches service-account JSON, `*.iam.gserviceaccount.com` emails, `private_key_id`, JWT-shaped bearer tokens, publicized server secrets (`NEXT_PUBLIC_*SERVICE_ACCOUNT/PRIVATE_KEY/CREDENTIAL`), and password fields in server/persistence/evidence code, while treating only the exact documented public Firebase web config name as public metadata; client-bundle scanner rejects `firebase-admin`, service-account, and private-key markers. All gates pass on the real build with 265 files and 50 reachable commits inspected.
- Unconfigured and partially configured builds succeed; production server serves `/` (200) and `/api/health` (`{"status":"ok",...}`) with every Firebase variable absent, keeping the guest workflow Firebase-independent.

## Decisions Made

- Kept `.nvmrc` at 24 (CI/workstation parity) with `engines.node >=22.0.0` as the minimum reviewed floor Firebase Admin 14 requires.
- Fixed the moderate `npm audit` findings introduced by firebase-tools/firebase-admin via semver-compatible `overrides` (`@opentelemetry/core ^2.8.0`, `uuid ^11.1.1`) rather than accepting vulnerabilities or downgrading the CLI.
- `test:firebase` was added in Task 1 but joined the `quality` chain only in Task 2, after `firebase.json` made the demo emulator real — matching the plan's "only after a demo-project emulator is configured" sequencing.
- Password reset intentionally reports success for unknown accounts (with local cooldown) so reset cannot enumerate registrations.
- A link attempt returning a different UID signs the session out and reports `link_unavailable` — progress can never silently migrate across UIDs.
