## Accomplishments

- `src/server/firebase-admin.ts` initializes Admin lazily only when the complete cloud configuration exists (flag exactly `"true"`, plausible project id, and either a structurally validated in-memory service-account shape matching the project or both emulator hosts); importing the module never loads the SDK, and every failure collapses into a single `CloudUnavailableError` that carries no value or provider detail.
- `src/server/firebase-identity.ts` accepts exactly one bounded `Bearer` Authorization value (missing, duplicate — comma-joined —, mixed-scheme, oversized, and malformed inputs all fail closed), verifies the token for the configured project, requires `email_verified === true` provider-independently (so unverified email/password identities are rejected while verified password and Google identities share one wrapper/profile path), and returns only a frozen `{ kind, uid }` wrapper with a path-safe UID shape. Nothing is ever logged from the verification path (asserted by tests).
- `src/server/progress-repository.ts` methods accept the verified identity wrapper — never raw client strings — and build the fixed `learningProfiles/{uid}` path internally inside Firestore transactions: strict tolerant DTO reads, monotonic one-time `local_import` (forced provenance re-tagging + 409-on-replay marker), idempotency-identifier replacement, a 50-attempt cap with explicit oldest deletion, and same-user document deletion.
- `src/server/progress-service.ts` computes the SHA-256 idempotency identifier from the internal UID and bounded numeric outcome only (no learner content), derives the approved lesson server-side from the unique challenge→(project, skill)→lesson registry mapping, and exposes a never-throwing `syncAssessment` returning only `cloud_saved | local_only | unauthorized | cloud_unavailable`.
- `/api/progress` GET/POST/DELETE are same-origin (Origin must match Host when present; opaque origins fail closed), rate-limited, `no-store`, exact-key, and re-authorized at the data boundary; unauthenticated or forged access is 401, infrastructure loss is a safe retryable 503, and no client field (uid, email, provider, path, provenance, completion, score) survives the strict schemas.
- The public assess route persists the bounded `server_verified` summary only for a valid verified identity **after** `assessChallengeWorkflow` decides the exact response; invalid/unverified/unavailable identity or Firestore failure produces no write and only a bounded degraded sync fact while the authoritative assessment, report, and local progress remain intact (proved with the cloud feature disabled mid-suite in the emulator tests and with the credential-free build serving all existing routes).
- 64 new tests: 44 fake-service unit tests (config shapes, header adversaries, wrapper minimality, isolation, replay, retention, content-minimal documents) plus 8 route-boundary tests and 12 emulator integration tests (unverified denial, verified password/Google same-contract success, restart restoration, two-user isolation, concurrent replay, retention, single-collection content-minimal documents, deletion, failed-assessment non-completion, import provenance, disabled-Firebase degraded operation). Full suite: 258 unit/route tests, 23 emulator tests, production build, and both leakage scanners green.

## Decisions Made

- **Provider-independent email verification:** rather than branching on the token's sign-in provider (which would put a password-provider literal inside server code guarded by the password-boundary scanner), the DAL requires `email_verified === true` for every identity. This is strictly fail-closed, rejects exactly the unverified email/password case the plan targets, and passes verified password and Google identities through the identical wrapper path.
- **`challengeSource: "prevalidated"` for persisted summaries:** the assess request carries no challenge source and the plan forbids accepting `source` from a progress-write client; the deterministic assessment authority is always the prevalidated fixture boundary, so the server records that fact.
- **Lesson difficulty from the registry:** each curated challenge's (projectId, targetSkill) pair maps to exactly one lesson, whose registry difficulty is authoritative for stored summaries — the client cannot vary the recorded difficulty of a guided attempt.
- **Single-document model:** the profile (completions + capped attempts + import marker) lives in one `learningProfiles/{uid}` document mutated transactionally, which makes replay/concurrency collapse, retention, isolation, and deletion trivially verifiable and keeps rules-denied client access aligned with the existing rules suite.
- **`cloudSync` as optional shared-schema field:** the strict client parser (`assessmentResponseSchema.safeParse`) continues to accept previously saved assessments while new responses carry the bounded sync fact; it is descriptive only and cannot change completion/test authority.
