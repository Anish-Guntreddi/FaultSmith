# Phase 01.1: Personalized Learner Accounts, Cloud Progress, and Metrics Dashboard - Context

**Gathered:** July 19, 2026
**Status:** Ready for planning
**Source:** PRD Express Path (`docs/PERSONALIZED_LEARNING_PRD.md`)

<domain>
## Phase Boundary

Deliver a guest-first My Progress dashboard and optional verified email/password or Google-authenticated cross-device synchronization through Firebase Authentication and Cloud Firestore. The Next.js server verifies identity and mediates cloud learning-data writes. The existing anonymous browser-local experience, validated fixture fallback, deterministic completion authority, hidden-answer containment, and no-host-execution boundary remain intact.

This phase includes credential-free domain/UI work, Firebase emulator integration, independent QA/security review, and an approval-gated real Firebase/Netlify preview proof. It does not add instructor features, grading, certification, raw learner-content storage, or mandatory authentication.

</domain>

<decisions>
## Implementation Decisions

### Access and identity

- Guest access remains the default; no login wall may guard the application, roadmap, challenges, reports, health route, or judging workflow.
- The three access paths are **Continue as guest**, **Create account / Log in with email and password**, and **Continue with Google**.
- Email/password and Google both resolve to the same UID-keyed learning-profile and metrics contracts; Firebase Authentication is the only credential authority.
- New email/password accounts must verify their email before any cloud progress route is available. The learner continues locally while verification is pending.
- Password reset and verification resend use Firebase email actions, generic states, cooldowns, and bounded retries. FaultSmith never stores, logs, forwards to its server, or adds password material to evidence.
- Provider collisions never trigger a silent merge or cross-UID copy. Optional linking requires an authenticated recent session, preserves the existing Firebase UID, and ships only if emulator and real-provider proof are green; otherwise the learner receives safe existing-method guidance.
- Signing in enables synchronization; signing out returns to guest/local mode without deleting cloud data.
- Authentication failure, cancellation, popup blocking, Firebase absence, and quota failure cannot block a challenge or report.
- A learner can explicitly delete cloud learning data; deletion cannot silently claim success.

### Progress and evidence authority

- Existing validated local progress remains the immediate persistence and reliability path.
- Only a deterministic `verified` assessment may complete a lesson.
- Failing, abandoned, test-only, or unsubmitted work cannot create completion.
- Cloud writes are idempotent and server-mediated after Firebase ID-token verification.
- Imported local completion is labeled `local_import`; future assessment persistence is labeled `server_verified`.
- Local/cloud merges are monotonic and cannot erase newer verified progress.
- Cloud history is capped at 50 attempt summaries and nine lesson completions.

### Personal metrics

- My Progress works in local-only and synced modes.
- Display completion, phase status, bounded score dimensions, hints, independent-solve rate, test-run process evidence, strongest practiced skill, reinforcement priority, recent history, and a deterministic next recommendation.
- Test-run count is descriptive and must never penalize experimentation.
- Every recommendation displays a plain-language reason.
- Metrics are practice evidence, not grades, certification, ranking, or formal mastery claims.

### Data minimization

- Firestore may store only strict learning-profile and attempt-summary contracts.
- Cloud data excludes source code, learner hypotheses/explanations, hint text, hidden answers, raw test output, prompts, provider IDs, auth tokens, credentials, names, and duplicated email addresses.
- Firebase Authentication may retain provider identity and password credentials under Firebase's account record; FaultSmith does not copy them into Firestore, localStorage, logs, DTOs, or evidence.
- Logs and evidence contain bounded outcome/timing only and no UID or internal document path.

### Firebase boundary

- Netlify hosts the Next.js application; Firebase supplies Authentication and Cloud Firestore only.
- Use a Firebase Spark project; no billing upgrade, Cloud Functions, Storage, Hosting, BigQuery, AI Logic, paid backup/PITR/clone/TTL, or unrelated Firebase product is authorized.
- The browser uses public Firebase web configuration only for Authentication.
- The server verifies Firebase ID tokens and uses server-only Firebase Admin access for Firestore.
- Direct browser Firestore access is denied through deployed Security Rules.
- Firebase initialization is lazy and configuration-gated; partial configuration fails to `local_only` or `cloud_degraded` without crashing build/startup.
- Current Firebase Admin Node.js support requires a Node 22-or-newer runtime; local, CI, and Netlify runtime choices must align.

### API and trust boundary

- Add authenticated progress read/import/delete routes with strict Zod contracts and `Cache-Control: no-store`.
- Existing assessment persists a bounded attempt only after deterministic policy completes; model output never promotes failing evidence.
- The server derives UID only from a verified token and accepts no client UID or Firestore path.
- Missing identity affects progress routes only and never changes anonymous challenge routes.
- Authorization values are bounded; invalid, expired, wrong-project, malformed, and oversized tokens fail closed with sanitized responses.

### Security and deployment

- Service-account material remains server-only and absent from source, Git history, logs, evidence, screenshots, and client bundles.
- Enable a required Firebase password policy and email-enumeration protection; application behavior cannot depend on specific email-exists/user-not-found errors.
- Only reviewed localhost/Netlify origins may appear in Google authorized domains and verification/reset continue URLs.
- CSP permits only exact Firebase/Google origins empirically required by the approved sign-in flow; wildcard expansion is forbidden.
- App Check can begin in observation mode, but enforcement waits for proven local, preview, and mobile compatibility.
- Netlify/app rate controls cover progress/auth surfaces; paid OpenAI paths retain separate stricter shared/edge controls.
- The real Firebase project and Netlify preview are human credential/deployment gates; no agent may create or mutate them without explicit authorization.

### Reliability and rollback

- Local state updates before cloud sync; cloud latency never delays the report.
- Sync retries are bounded and idempotent.
- Disabling `NEXT_PUBLIC_FAULTSMITH_CLOUD_PROGRESS` and removing server Firebase credentials returns the same build to local-only behavior.
- The validated fixture fallback is independent of Firebase state.
- If cloud identity, isolation, CSP, preview, or full gates miss the July 20 cutoff, ship the local personalized dashboard with cloud mode disabled.

### Validation

- Use Firebase Authentication and Firestore emulators with a demo project ID for credential-free integration tests.
- Prove same-user success, cross-user denial, direct-client Firestore denial, invalid token rejection, idempotency, retention, deletion, merge behavior, and degraded fallback.
- Add guest, email signup/login/verification/reset, Google sign-in, provider collision/linking fallback, cancellation, failure, sign-out, cross-session restore, keyboard, axe, 1440×900, and 390×844 browser coverage.
- Run lint, typecheck, unit/integration, build, client-bundle/source/history security, dependency audit, Playwright/axe, fallback smoke, production smoke, and primary-demo gates.
- Require independent QA/accessibility and security/adversarial review against one candidate SHA and self-heal accepted findings.

### Documentation

- Update PRD, GSD requirements/roadmap/state, README, BUILD_LOG, ROADMAP, THREAT_MODEL, TESTING, DEPLOYMENT, DEMO_SCRIPT, SUBMISSION, and COMPLETION_REPORT with objective evidence.
- Evidence must distinguish local dashboard, emulator proof, credentialed Firebase proof, OpenAI live proof, fixture fallback, and Netlify preview/production proof.

### Codex's Discretion

- Exact component boundaries and visual layout within the established FaultSmith design system.
- Exact Firestore collection/document names, provided verified UID ownership and strict bounded schemas are preserved.
- Exact metric selector implementation and chart-free presentation, provided formulas remain deterministic, explainable, and tested.
- Whether progress import is a dedicated endpoint or a one-time server action, provided provenance, idempotency, and validation requirements are met.
- Exact service-account environment encoding compatible with Netlify, provided it remains one server-only secret and passes scanners.
- Exact test decomposition and wave parallelism, provided credential-free work precedes human-gated provider/deployment proof.

</decisions>

<specifics>
## Specific Ideas

- Use the existing nine lessons and existing score/hint/test-run fields as the initial dashboard substrate.
- Prefer compact evidence cards and phase progress over a generic analytics console.
- Label storage state as **On this device**, **Verification required**, **Syncing**, **Synced to account**, or **Saved on this device—cloud unavailable**.
- Keep the zero-token beginner story visible in the dashboard and demo.
- Preserve the primary Expense Approval boundary-condition flow and under-three-minute recording narrative.
- Firebase Spark's currently documented free quota is adequate for submission testing, but usage must be bounded and monitored rather than described as unlimited.

</specifics>
