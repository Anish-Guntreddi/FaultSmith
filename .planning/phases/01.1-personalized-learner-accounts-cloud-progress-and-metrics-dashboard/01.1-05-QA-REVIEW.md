## Findings

### QA-01 — Lint warning: unused `passwordBoundaryRules` import; password-boundary scanner rules have no direct unit coverage (Severity: Low)

- **Where:** `scripts/check-source-security.test.mjs:12` imports `passwordBoundaryRules` from `scripts/check-source-security.mjs` but never exercises it.
- **Reproduction:** `npm run lint` → `12:3 warning 'passwordBoundaryRules' is defined but never used @typescript-eslint/no-unused-vars` (exit 0).
- **Why it matters:** (a) it is the only warning in an otherwise clean gate, normalizing warning noise; (b) more substantively, the `password-boundary` scanner rule — a load-bearing Phase 01.1 security control (AUTH-03/PRIV-01 enforcement) — has no direct unit test proving it matches password material in server paths and respects its allowlist. The import without a test suggests intended-but-missing coverage. Pre-existing since Plan 01.1-02 and logged in `deferred-items.md` explicitly as "candidate cleanup for Plan 01.1-05".
- **Expected disposition:** Add a real test that exercises `passwordBoundaryRules` (removing the warning by using the import, not by deleting it), preserving scanner authority.

### QA-02 — Inert "Continue as guest" control gives no activation feedback (Severity: Low, shared with product review PROD-01)

- **Where:** `src/components/progress-sync.tsx` signed-out summary mode.
- **Reproduction:** Keyboard-focus the "Continue as guest" button and activate it: no state change, no `role="status"` announcement, nothing for a screen reader to confirm. All neighboring buttons produce visible + announced outcomes.
- **QA angle (distinct from product angle):** an interactive element whose activation produces no perceivable result is an accessibility defect pattern (silent control). Severity stays Low because guest mode genuinely is the default and no data or flow is harmed.
- **Expected disposition:** Activation should produce announced confirmation (and reasonably dismiss any open account form).
